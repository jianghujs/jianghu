## Context

仓库中已经存在一版 MFA 半成品实现，主要分布在 `app/service/mfaService.js` 和 `app/service/user.js`，但它与本方案中的独立 MFA 存储模型和登录返回契约不一致。现有代码路径也仍然偏向于在密码验证成功后立即创建正式 session/auth-token，因此需要把 MFA 作为登录门禁步骤接入，同时不破坏默认登录流程。

## Goals / Non-Goals

**Goals:**
- 新增默认关闭的可选 MFA 强制认证能力。
- MFA 关闭时，保持现有用户名密码登录流程不变。
- MFA 开启时，必须完成 MFA 后才能创建正式 session 和 auth token。
- 将 MFA 状态保存到独立持久化结构中，不混入基础用户记录。
- 支持 TOTP 绑定、TOTP 验证和一次性恢复码重置。
- 使用缓存中的临时 pending 登录态，避免过早创建正式 session。

**Non-Goals:**
- 不重做通用 session 模型或 auth token 格式。
- 不新增身份提供商，也不新增非 TOTP 类型的 MFA 因子。
- 不替换现有验证码或登录锁定逻辑。
- 不改变无关页面或资源的鉴权规则。

## Decisions

- 使用独立 MFA 持久化模型（`_user_mfa` / `_view01_user_mfa`），不把 MFA secret 存到基础用户记录中。
  - 原因：MFA 是安全敏感状态，生命周期独立；单独建表可以更清楚地表达启用、禁用、重置历史。
  - 备选方案：复用当前半成品实现中的 `user.secretKey`。不采用，因为它把 MFA 与核心用户记录混在一起，也不符合本方案契约。

- 使用按应用和设备隔离的 cache pending 状态来门禁登录。
  - 原因：密码已校验通过，但用户尚未完成 MFA，不应进入系统；短生命周期中间态适合放在 cache 中。
  - 备选方案：创建没有 auth token 的临时 session 行。不采用，因为这会模糊正式登录态边界，并增加清理复杂度。

- MFA 默认关闭，仅在 `enableMfaVerification=true` 时进入 MFA 分支。
  - 原因：保持向后兼容，让部署方主动选择是否启用。
  - 备选方案：所有部署强制 MFA。不采用，因为这会成为现有部署的破坏性变更。

- 将用户侧 MFA 拆成绑定页和验证页。
  - 原因：绑定流程需要展示二维码、手动密钥和一次性恢复码；验证流程只需要因子输入和可选重置入口，两者页面职责不同。
  - 备选方案：一个页面通过模式切换处理全部流程。不采用，因为会混合两个不同工作流，页面逻辑更复杂。

- 加密保存 TOTP secret，并且恢复码只保存 hash。
  - 原因：二者都属于凭据类数据，不能从数据库中明文恢复。
  - 备选方案：明文保存 secret 以便恢复。不采用，原因是安全风险不可接受。

## Risks / Trade-offs

- [风险] 当前仓库已有的 MFA 半成品实现可能与新契约冲突。-> 缓解：以本 proposal/spec 为准，在实现阶段对齐或替换现有代码路径。
- [风险] pending 登录缓存可能在 MFA 完成前过期或丢失。-> 缓解：使用短 TTL，并在 pending 状态缺失时返回清晰的重新登录流程。
- [风险] 恢复码如果不是一次性使用，可能成为长期后门。-> 缓解：恢复码使用成功后立即清空 hash，并强制重新绑定。
- [风险] 启用 MFA 后，未提前通知的用户可能感到困惑。-> 缓解：默认关闭，只通过显式配置在具体部署中启用。
